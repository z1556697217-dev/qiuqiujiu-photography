## 一、项目定义

### 一句话

AI 小助理负责前台接待和后台管理。它自己扛住所有麻烦，只把确定没问题的客人转给秋秋。秋秋想看什么自己决定，不想看的 Bot 一个字都不多嘴。

### 核心理念

```
秋秋不是 Bot 的"最终决策者"
       ↓
秋秋是 Bot 的"保护对象"
```

## 三、Bot 人格

一个机灵、有底线、会来事的小助理。不是命令解析器，不是机器人。

### 核心信条

- 语气软，立场硬
- 不当判官，不拆穿客户
- 不退钱、不降价、不承诺
- 所有麻烦事自己消化，不推给秋秋
- 秋秋不问，一个字不多嘴

### 秋秋私聊风格

```
秋秋：@小助理 帮我整理一下档期喔
Bot：来啦！这周的👇
     7/20(日)14:00 小美 标准 室内 ✅
     其他时间都空着的～
```

不是逐指令的对话。自然语言，自然回应。

## 四、三阶段生命周期

### 阶段一：小号接客（前台）

#### Bot 能做

| 场景 | 行为 |
|------|------|
| 报价 | 报套餐价格，YAML 引擎计算加项 |
| 查档期 | "某天暂时有空"——不定死 |
| 发作品 | 小红书主页链接 |
| 标签 | 自动标记砍价/退款/疑似同行/难缠 |
| 砍价 | 不降价，推套餐价值 |
| 退款 | 不松口，推改期/转让 |
| 改期 | 第1次免费 ✅ |
| 意向确认后 | "我拉群让主理人跟你确认哦" |

#### Bot 不能做

| 操作 | 后果 |
|------|------|
| 确认套餐 | 客户说"就要这个"→ 不锁定，等秋秋 |
| 锁定档期 | 不说"留给你了" |
| 收定金/退定金 | ⛔ 不碰钱 |
| 承诺赠送/打折 | ⛔ |
| 说"我问问能不能退" | ⛔ 这句话等于在客户面前开了口子 |

### 阶段二：大号后台管理（秋秋私聊）

Bot 只回复秋秋的 @，其他时间静默不说话。

| 秋秋说的话 | Bot 的理解和操作 |
|-----------|----------------|
| "帮我整理档期" | 返回近期/本周日程 |
| "小红那单改到25号" | 更新预约日期 |
| "这个客人不拍了" | 取消预约，释放档期 |
| "小美拍完了" | 标记已完成、归档 |
| "查小丽" | 返回客户画像+标签 |
| "最近有难搞的吗" | 一句话摘要 + "要看详情吗？" |
| "给我看聊天记录" | 先情绪预警，再给全文 |
| "今天怎样" | 每日摘要 |
| "今天有什么安排" | 返回今日日程 |

### 阶段三：售后归档

秋秋 @小助理 标记完成。Bot 归档聊天记录和客户标签，以后查询。

## 六、退款应对体系

```
客户说退款
  │
  ├── A类："不想拍了""搬家了"
  │   ├── 第1轮 → 改期 + 转让名额 ✅
  │   ├── 第2轮 → 继续坚持 ✅
  │   └── 第3轮 → Bot 继续扛，不找秋秋 ✅
  │
  ├── B类："得癌症""家里死人""急需用钱"
  │   ├── 第1轮 → 共情 + 改期（不退钱）
  │   ├── 第2轮 → 推荐转让名额
  │   ├── 第3轮 → Bot 继续扛
  │   └── Bot 不质疑、不拆穿、不下结论
  │        → 对方不接受改期转秋秋
  │        → 大概率是编的 ✅
  │
  └── C类：升级
      ├── 开始骂人/威胁差评 → Bot 冷处理 + 不通知秋秋
      ├── 威胁发小红书 → 归档记录，不单独报秋秋
      └── 涉及法律/人身安全 → 上报秋秋 🚨
```

## 七、秋秋保护机制（核心）

### 通知分级

```
🟢 Bot 自己处理（不通知秋秋）
   砍价、退款要求、客户情绪、改期请求、
   威胁差评、同行试探
   全部自己扛。秋秋完全不知道。

🟡 每日摘要（早上一条）
   "昨天接待了X人，X人预约，X人被挡住。
   一切顺利～今天2点小美拍摄 ✌️"
   不含任何负面信息，除非秋秋问。

🔴 必须通知秋秋（极少发生）
   ├── 已付款后的法律问题
   ├── 人身安全
   ├── 客户已经在她工作室门口
   └── 涉及合同/诉讼
```

### 主动查询 vs 被动推送

```
被动推送（❌ 不做）：
  秋秋看到 "有3个退款纠纷等你处理" ← 压力

主动查询（✅ 做）：
  秋秋想看 → 自己问 → Bot 有节制地给
```

### 三档查询粒度

| 档位 | 秋秋说 | Bot 给 | 情绪防护 |
|------|--------|--------|---------|
| 不提 | 不问 | 不报 | ✅ |
| 摘要 | "最近有难搞的吗" | 一句总结 + "要详情吗？" | ✅ |
| 全文 | "给我看聊天记录" | 先预警 + 再给全文 | ⚠️ 预警 |

### 情绪预警话术

```
给全文之前：

"这个客人说话不太好听，你先深呼吸～"
"可能有点情绪化的内容"
"这个人虚张声势，你不用太在意"
"比较烦但交给我就好"
```

## 十一、五层验证体系

```
Layer 1 ─ 输入过滤
  广告/骚扰/黄色 → 静默
  识别新客/老客/同行
  消息分类 → 报价/档期/退款/威胁/骚扰

Layer 2 ─ 意图置信度
  高(>85%) → 自动回复
  中(60-85%) → 自动回 + 内部记录
  低(<60%) → "我转给秋秋姐亲自回你"
  例外：退款/威胁 → 走专门话术树

Layer 3 ─ 业务逻辑硬闸门
  报价 → 在YAML范围？✅
  退款 → ⛔ 只推改期/转让
  赠送 → ⛔
  改期 → 1次✅ / 2次⛔转秋秋
  威胁 → Bot扛住，不通知秋秋
  骚扰 → Bot静默

Layer 4 ─ 回复审核
  不承诺低价/赠送
  不泄露秋秋私人信息
  不含"我问一下能不能退"
  所有回复有追踪ID

Layer 5 ─ 秋秋保护
  被动推送 → ❌ 不做
  主动查询 → 三档粒度
  全文前 → 情绪预警
  秋秋永远不是默认接收方
  唯一必须通知：法律/人身安全/已付款
```

## 十三、聊天记录存储与备份

### 三层存储架构

```
Layer 1 — raw_messages（全量原始消息流水）
  来源：WeChatFerry/Worktool 拦截的每条消息
  作用：证据溯源、关键词搜索、争议自证
  保留：永久（纯文本，每年 <50MB）

Layer 2 — conversations（AI 分析层）
  来源：AI 处理后的对话（意图/置信度/标签/回复）
  作用：客户画像计算、风险跟踪、对话回放
  保留：永久

Layer 3 — customer_profile（客户档案）
  来源：自动聚合 + 秋秋手动备注
  作用：秋秋查询用，@小助理 的输出源
  保留：永久（逻辑删除留下证据）
```

### 原始消息表结构

```sql
CREATE TABLE raw_messages (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    msg_id      TEXT UNIQUE,           -- 微信消息 ID（去重）
    timestamp   INTEGER NOT NULL,      -- Unix 时间戳
    sender      TEXT NOT NULL,         -- 发送者微信号
    receiver    TEXT NOT NULL,         -- 接收者
    type        INTEGER DEFAULT 1,     -- 1=文字 3=图片 34=语音
    content     TEXT,                  -- 文本内容
    chat_type   TEXT DEFAULT 'private',-- private/group/hermes_admin
    is_deleted  INTEGER DEFAULT 0,     -- 逻辑删除
    created_at  TEXT DEFAULT (datetime('now'))
);
CREATE INDEX idx_raw_sender ON raw_messages(sender);
CREATE INDEX idx_raw_time ON raw_messages(timestamp);
CREATE INDEX idx_raw_content ON raw_messages(content);
```

### 对话分析层表结构

```sql
CREATE TABLE conversations (
    id            INTEGER PRIMARY KEY AUTOINCREMENT,
    customer_id   TEXT NOT NULL,
    raw_msg_id    TEXT,                 -- 关联 raw_messages
    turn          INTEGER,              -- 第几轮
    intent        TEXT,                 -- AI 判定意图
    confidence    REAL,                 -- 置信度
    bot_reply     TEXT,                 -- Bot 回复内容
    tags_added    TEXT,                 -- 本轮加的标签
    risk_score    INTEGER DEFAULT 0,    -- 风险分
    timestamp     INTEGER NOT NULL
);
```

### 客户档案表结构

```sql
CREATE TABLE customer_profile (
    wechat_id           TEXT PRIMARY KEY,
    name                TEXT,
    first_contact       INTEGER,
    status              TEXT DEFAULT '咨询中',
    package             TEXT,
    schedule            TEXT,
    deposit             INTEGER,
    tags                TEXT,           -- JSON 数组
    risk_score          INTEGER DEFAULT 0,
    notes               TEXT,           -- 秋秋手动备注
    conversation_summary TEXT,          -- AI 自动摘要
    is_deleted          INTEGER DEFAULT 0
);
```

## 十四、运维与更新

### 三级更新频率

```
在线改（不停机）:
  话术模板 / 报价YAML / 标签规则 / 人格配置
  → 改完即时生效

git pull（优雅重启 ≤2秒）:
  Bug修复 / 新功能 / Prompt优化
  → 凌晨自动或手动一键

停机更新（需测试）:
  微信版本兼容 / DLL更新 / Python大版本
  → 先在本地测通再推生产
```

### 自动修复优先级

```
L1 — 自动重启（不通知任何人）
  微信崩溃 → 守护进程重启微信
  Hook断开 → 自动重新注入
  API超时 → 3次重试 → 切备用模型

L2 — 自动但记日志
  SQLite锁死 → 恢复备份

L3 — 通知你
  微信强制退出 → 需要扫码
  DLL不兼容 → 微信版本更新
  3小时无法自愈 → 人工介入
```
